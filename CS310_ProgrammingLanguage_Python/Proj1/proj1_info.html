Project 1

Goal

The "manta" of this class is that:

  • There are programmers who design and implement languages;
  • There are users who use that language to complete their tasks.

Further, we say that:

  • A programming language is the syntactic sugar that makes some underlying maths palatable to users .
    • Where that sugar has three parts:
      • Succinct structures (so programmer can code one thing in less space);
      • Shortcuts (so users can specify repeated things in less time);
      • Watchdogs that alert us to possible problems.
  • That the better we understand the maths, the better we can design a language.
    • E.g. our watchdogs can refer to the maths to find some problems.

The goal of this project is to

  • Show students aspects of succinct, shortcuts, watchdogs, maths using a simple engine for text adventure games;
  • All the while, learning some basic Python coding techniques.

Caution to Advanced Students

This is a simple introductory exercise for beginners to language design. By the end of the semester, we will be using our own LISP interpreter (built in this class) to implement recursive functions, closures, nested name spaces, and lots of other good stuff besides. But that is for later. For now, we just want to make sure our group members are all walking at the same speed, before we start running.

Not Succinct Enough for Users

Here are some user-level annoyances in the above:

  • The full name of states have to be given, every time they are used.
    • WANTED1: some partial match so you only have to type the first N unique letters of a state to identify it.
    • WANTED2: a warning if the typed letters match nothing or more than one thing.
  • Having to add text like \n what to do next?.
    • WANTED3: these should be auto-added.
  • Opening brackets have to matched. Can we do better with some other syntax? e.g.
    • WANTED4:
      • Every node is one paragraph (paragraphs are separated by one blank line).
      • Every line not starting with > is a description.
      • Every line starting with > followed by a state name and a question is a link to another state.

Of course, that surface form would have to be read into some structure. WANTED5: the following internal class structure.

Edges:

class Edge:
    def __init__(self,here,there,txt):
      self.description = txt     # why am i making this jump? 
      self.here        = here    # where do i start
      self.there       = there   # where to i end
      self.here.out   += [self]  # btw, tell here that they can go there

    def __repr__(self):
      return "E(" + self.here.name + " > " + self.there.name + ")"

Nodes:

class Node:
  end   = "."
  start = "!"

  def __init__(self,g,id,name,stop=False,start=False):
      self.id = id
      self.graph = g          # where do i live?
      self.name = name        # what is my name?
      self.description = ""   # tell me about myself
      self.stop = stop        # am i a stop node?
      self.start = start      # am i a start node?
      self.out = []           # where do i connect to

  def also(self,txt):
      "adds text to description"
      sep = "\n" if self.description else ""
      self.description += sep + txt

  def __repr__(self):
      return "N( :id " + str(self.id) + \
             "\n   :name " + self.name + \
             "\n   :about '" + self.description + "'" + \
             "\n   :out " + str(self.out) + ") "

Graphs:

class Graph:
    def __init__(self):
      self.nodes = []    # nodes, stored in creation order
      self.keys  = {}    # nodes indexed by name
      self.m = None      # adjacency matrix
      self.mPrime = None # transitive closure matrix

    def node(self,name):
      "returns a old node from cache or a new node"
      if not name in self.keys:
        self.keys[name] = self.newNode(name)
      return self.keys[name]

    def newNode(self,name):
      " create a new node"
      id = len(self.nodes) 
      tmp = Node(self,id,name)
      tmp.start = Node.start in name
      tmp.end   = Node.end   in name
      self.nodes += [tmp]
      return tmp

The general problem with the above is that if we change the nodes and edges, then we have to change the code:

  • The start and end nodes should not be hard-wired into the interpreter:
    • WANTED6: it should be inferred from the specification- it
  • The number of stages in the code should not be hard-wired into the code (as done above).

Watchdogs

So here are the watch dogs we can apply to the island matrix. If any of the following are false, then report an error:

  • There is one and one only start node.
  • There is one and one only stop node.
  • Start nodes have no incoming edges.
  • Stop nodes have no outgoing edges.
  • All nodes should connect downstream to the stop node.
  • The start node should connect to all other nodes.
  • There should only be one island.
  • All internal nodes (that are not start or stop nodes)
    • Need at least one in-coming edge.
    • Need at least one out-going edge

Note that all of these are queries to the island matrix. There are two ways to do that:

  • The dumb, slow way
    • Go to every node and walk upstream and downstream to see what you can find.
  • The smarter way:
    • Compute at one go, a matrix showing what everything can reach.
    • Then query that matrix.

This brings us to transitive closure:

  • If Oregon touches Idaho and Idaho touches Utah
  • Therefore, transitively, I can travel from Oregon to Utah.

Consider the adjacency matrix for:

It would be:

 M=
 0100
 0001
 0000
 1010

Now write everywhere you can get to from everywhere else. Note that from c, you can't go anywhere (so the transitive closure of c should be empty). But a,b,d are in a loop so from any of them, we can go anywhere else:

 M' =
 1111
 1111
 0000
 1111

The transitive closure algorithm says for an adjacency matrix M with indexes 0..N, if can get from i to k and k to j then I can go i to k:

for k in range(n):
  for i in range(n):
     for j i range(n):
        M[i][j] = M[i][j] or (M[i][k] and M[k][j])

Note that the inputs and outputs of the above are all adjacency matrices
